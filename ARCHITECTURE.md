# Kinly — Architecture Overview

Kinly is a React Native / Expo app for small private accountability circles ("Growth Circles" of 2-10 friends). This doc maps each screen to what it does and how it's wired up, so future work can find the right file fast.

## Stack

- **Navigation**: `@react-navigation` — root native-stack (`Onboarding`, `Main`, `CircleSettings`, `EditProfile`) wrapping a bottom-tabs navigator (`Today`, `Circle`, `Goals`, `Connection`, `Profile`). See [src/navigation/types.ts](src/navigation/types.ts), [src/navigation/MainTabs.tsx](src/navigation/MainTabs.tsx).
- **State**: Zustand `useAuthStore` holds the signed-in `user` and `activeCircleId` (a user can belong to multiple circles but only one is "active" at a time). `activeCircleId` is persisted to `AsyncStorage` via `zustand/middleware`'s `persist` (`partialize`d to just that one field — `user`/`sessionLoading` deliberately stay unpersisted, since `useBootstrapSession` already re-derives `user` from the live Supabase session every launch, and persisting it too would risk showing stale data instead of a loading state). This used to be in-memory only, which reset it to `null` on every cold start — `RootNavigator` would then fall through to `OnboardingScreen`'s `CircleStep`, whose "no active circle yet" fallback effect auto-picks `circles[0]` (oldest circle, by insertion order), silently switching a returning multi-circle user back to their original circle on every app restart with no way to tell it had happened short of checking Circle Settings. `hasHydrated` (set via `onRehydrateStorage`) gates `RootNavigator`'s loading screen alongside `sessionLoading`, so it can't evaluate `readyForMain` off a not-yet-rehydrated `activeCircleId` and flash Onboarding before the real value loads from disk.
- **Server state**: TanStack React Query, one hook file per domain in `src/hooks/`, persisted to `AsyncStorage` for offline-first caching ([src/lib/persister.ts](src/lib/persister.ts)).
- **Backend**: Supabase — Postgres + RLS, Auth (email/password + Google OAuth), Storage (avatars, vision-board images), Edge Functions (nudge messages, notifications, weekly recap, circle AI, streak-at-risk cron).
- **Linting**: [eslint.config.js](eslint.config.js), set up via `npx expo lint` (the official Expo flow — `eslint-config-expo`, flat config), not hand-rolled. Previously the project had no linting at all, only `tsc --noEmit` — the first real run immediately caught 3 genuine unused imports and several `react/no-unescaped-entities` issues (raw apostrophes/quotes in JSX text, fixed by wrapping the string in a `{'...'}`/`` {`...`} `` JS expression rather than HTML-entity-escaping, which is more readable in a codebase with no other precedent for entities). `npm run lint` runs it.
- **Theme**: static tokens in [src/theme/colors.ts](src/theme/colors.ts) — `colors`, `categoryColors` (per interest pillar), `gradients`, `radii`, `shadow`. No dark mode yet (deliberately deferred). `textSecondary` is `#7A5C42`, darkened from an earlier `#9C7A5E` that only hit ~3.7:1 contrast against the app's light backgrounds — below WCAG AA's 4.5:1 minimum for normal text; found via a UI/UX audit (`.claude/skills/ui-ux-pro-max`), same warm-brown family, now ~5.7–6.1:1. Each semantic color (`success`, `celebration`) is deliberately a distinct hex from the category color it shares a hue family with (`health.solid`, `relationships.solid`) — same visual family, but not literally the same value, so they stay distinguishable if ever shown together. Palette is consolidated to orange + white/cream, with blue reserved as the single deliberate exception for the Learning pillar (`categoryColors.learning`) and for `gradients.inspiration` where it meets Ideas — this replaced an earlier palette that scattered a different hue per screen (navy `primary` on Onboarding, teal `secondary`/`gradients.growth` on Garden), which read as visually inconsistent. Because 4 of 5 categories now live in the same orange family, icons/emoji (not color) are the primary way to tell pillars apart at a glance; color is secondary reinforcement. `danger` stays red as the other deliberate exception, since an all-orange error state would lose urgency.
- **Illustrations**: custom SVG art lives in [assets/illustrations/](assets/illustrations/), imported directly as components via `react-native-svg-transformer` ([metro.config.js](metro.config.js), [svg.d.ts](svg.d.ts)) — the first raw `.svg` imports in the codebase; the app icon/`Logo` are still hand-coded `react-native-svg` primitives instead, unrelated to this pipeline. `GardenStageArt` ([src/components/GardenStageArt.tsx](src/components/GardenStageArt.tsx)) maps `GardenStage` (`seed`/`sprout`/`tree`/`bloom`) to real plant art, used by both `GardenCard` (per-member) and `GardenTeaser` (circle-level hero, picked by health tier). All 5 `GardenStage` values now have matching art (`wilted` included). Also used: `kinly-Goal.svg` (target icon) replaces the 🎯 emoji on goal tags/chips in `ConnectionScreen`; `kinly-idea.svg` (a more detailed bulb illustration than `idea-bulb.svg`) heads `GoalSuggestions`' "Suggested for you" section; `kinly-ill-buddy`, `kinly-ill-calendar-meetup`, `kinly-ill-rocket`, `kinly-ill-chat`, `kinly-ill-dice` replace the 🤝/📍/🚀/💬/✨ emoji in `BuddyCard`, `MeetUpCard`, `ChallengesCard`/`QuickActionsRow`, `DailyCircleCard`/`QuickActionsRow`, and the Connection "Light Moments" `DisclosureSection` header respectively (`DisclosureSection` now takes an optional `icon` prop).
- **Custom icon set** (`assets/icons/{mood,nudges,pillars,feed}/`): hand-drawn, single-color (`#F97316`) 24×24 stroke icons commissioned specifically to replace poorly-shaped platform emoji throughout the app — mood (happy/neutral/sad), nudges (cheer/water/walk/workout/study/streak), pillars (health/wealth/ideas/learning/relationships), and feed (camera/celebrate/chat/check/clock/delete/galaxy/lock/mail/masks/robot/rocket/sprout/wave). Every event/status type across the app now has a matching custom icon — no emoji fallbacks remain in `EVENT_STYLE` or elsewhere in the icon migration. `GoalsScreen.tsx` (streak/water-mark/photo-log/completed) and `QuickActionsRow`'s "Check In" action were missed in the original pass and still had raw 🔥/💧/📷/✅ emoji — caught and closed via a later UI/UX audit against `.claude/skills/ui-ux-pro-max`'s `no-emoji-icons` rule, reusing the same `StreakIcon`/`WaterIcon`/`CameraIcon`/`CheckIcon` already established elsewhere rather than commissioning anything new. Two integration patterns, same split as `TabIcons.tsx` vs. the raw illustration imports above:
  - **Static spots** (mood picker, nudge row, feed/event icons, card titles) use the raw `.svg` imports directly — `MoodCheckinCard`, `TodayScreen` (event feed icons, including `reminder`/`ask`/`challenge_completed`'s clock/chat/rocket icons, + nudge row + greeting wave), `FutureSelfCard`, `GuessWhoCard`, `VisionBoardCard`, `GardenCard`, `ConnectionScreen`'s delete button.
  - **Pillar icons need dynamic recoloring** (`InterestPicker`'s chips, `GoalsScreen`'s category picker, `CircleAICard`'s strongest/weakest pills) — selecting a chip flips its background to a solid category color, so a hardcoded-orange icon would go invisible on an orange chip. [src/components/icons/PillarIcons.tsx](src/components/icons/PillarIcons.tsx) reproduces all 5 as prop-driven `react-native-svg` primitives (`color` prop, same pattern as `TabIcons.tsx`) so they can flip to white when active. `CircleAICard`'s "Try: {challenge}" button reuses `IdeasIcon` in white for the same reason (solid-color button background).
  - **Gradient-card contrast**: `GardenTeaser` (title) renders white text on a colored gradient (`gradients.growth`), where the hardcoded-orange raw icons would have poor contrast. [src/components/icons/MonoIcons.tsx](src/components/icons/MonoIcons.tsx) reproduces `RobotIcon`/`SproutIcon` as prop-driven `color` components for this (same pattern as `PillarIcons.tsx`); `HealthIcon` from `PillarIcons.tsx` doubles as the water-droplet icon there too, since `nudges/water.svg` and `pillars/health.svg` are identical path data. `WeeklyRecapCard` used to be the other gradient case (`gradients.inspiration`, the one place blue leaked in via Learning) but was converted to the flat `cardShell` like everything else — `gradients.inspiration` is now unused and was removed from `theme/colors.ts` entirely rather than left as dead code. `MonoIcons.tsx` also has `CheckIcon` (`InterestPicker`'s selected-chip checkmark, white on a variable per-category solid color) and `HappyIcon`/`NeutralIcon`/`SadIcon` (`MoodCheckinCard`'s picker, which needs to flip between a muted gray and white depending on selection — a color range the raw hardcoded-orange mood `.svg` imports can't reach). `MoodCheckinCard`'s picker was redesigned to a specific reference: outlined/muted buttons by default, solid `colors.primary` fill only for the mood being tapped, rather than a uniform `colors.inputBg` fill for all three.
- **Brand**: two raster masters, both the same "two people, hands reaching" mark with photographic shading (kept raster rather than traced into SVG paths, which would lose the gradient/shading fidelity), used for different purposes because one color scheme doesn't read on both grounds:
  - [assets/brand/app-icon-master.png](assets/brand/app-icon-master.png) — solid white glyph on a flat orange field, purpose-built as an app icon. Drives every icon surface: `assets/icon.png` (opaque, iOS/general), `assets/android-icon-foreground.png` (transparent white glyph at 66% fill for Android's adaptive-icon safe zone) + `assets/android-icon-background.png` (solid `#F97316`, pinned to `colors.primary` exactly rather than the master's own sampled background pixel, which lands a shade more saturated — every icon surface needs to be pixel-identical in color) + `assets/android-icon-monochrome.png` (the same silhouette for Android 13+ themed icons), and `assets/favicon.png`.
  - [assets/brand/logo-master.png](assets/brand/logo-master.png) — the orange-to-amber gradient version on a white background, used anywhere the glyph needs to sit on a *light* surface and therefore needs its own color to read at all (a white glyph would be invisible there): `assets/brand/logo-web.png` (index.html's nav) and `assets/splash-icon.png` (shown against `colors.background` cream via `expo-splash-screen` — this was briefly wired to the white-glyph master and was invisible on cream until caught and switched to this one).

  [scripts/generate-brand-icons.mjs](scripts/generate-brand-icons.mjs) (uses `sharp`, installed on demand with `--no-save`, not a runtime dependency) does all of the above from both masters in one run. It chroma-keys each master's flat background to transparency by *distance from that master's own sampled background color* (white for `logo-master.png`, orange for `app-icon-master.png`) rather than a fixed "distance from white," snapping to a hard threshold with a narrow anti-aliasing band rather than a linear fade — a naive linear key fades light/low-saturation glyph colors toward transparent right along with the real background. Re-run the script if either master ever changes; `app.json`'s `adaptiveIcon.backgroundColor` and the `expo-notifications` plugin's icon `color` are set independently and need updating by hand if the brand color itself changes. [src/components/Logo.tsx](src/components/Logo.tsx) (hand-coded `react-native-svg` primitives, recolorable via `color`/`background` props) still renders the *previous* "friendly face" mark and is unrelated to this pipeline — used on Profile/EditProfile as an avatar-placeholder badge (a colored circle behind the mark), where a flat single-color primitive still reads fine at small sizes; not yet reconciled there. `OnboardingScreen`'s header was reconciled, though: it previously used `Logo.tsx` too, but a full-screen gradient header is exactly where the old/new mark mismatch was most visible, so it now renders `assets/brand/logo-white-glyph.png` instead — a one-time `sharp .trim()` crop of `android-icon-foreground.png` down to the glyph's own ink bounding box (the raw adaptive-icon layer has ~35% transparent safe-zone padding baked in, which would otherwise make the mark look small and off-center at a normal display size). Icon/adaptive-icon/splash changes are native-level resources baked in at build time — an `eas update` (JS-only OTA) will **not** update them on already-installed apps; only a fresh `eas build` + reinstall does. Most of `assets/brand/`'s other icons (`chat`/`heart`/`people`/`search`/`notification`/`circle_badge`/`minimal`) have the brand color hardcoded into the SVG rather than exposed as a prop, so they're only used in static single-color spots (e.g. `settings.svg` next to the Circle Settings link) — not the tab bar, which needs to dynamically tint icons gray/white (see `TabIcons.tsx` below). `assets/icons/` holds a separate, decoupled copy used only by `index.html`'s feature badges, so the website isn't coupled to the app's asset/build pipeline.
- **Platform status**: [app.json](app.json) is configured for both platforms and every EAS Update publish targets iOS + Android JS bundles, but no real iOS native build (`eas build --platform ios`) has been produced or tested yet — that requires an Apple Developer Program enrollment, which hasn't happened. `ios.infoPlist.NSPhotoLibraryUsageDescription` is set (needed for the avatar/vision-board image picker); revisit this section once a real iOS build is run, since native-config issues only surface there, not in `expo export` or `eas update`.
- **EAS Android build gotcha**: release builds (`preview`/`production` profiles) fail Gradle entirely if Sentry isn't fully configured — the `@sentry/react-native` Gradle plugin tries to upload source maps every release build and hard-fails (`error: An organization ID or slug is required`) since no real Sentry org/project is connected (Sentry is otherwise inert — see below, no DSN set). [eas.json](eas.json) sets `SENTRY_DISABLE_AUTO_UPLOAD=true` on all three build profiles to skip that step. Debug builds (plain `./gradlew assembleDebug`) never hit this, which is why local repro didn't initially catch it — only a real release-profile build does.
- **Status bar**: `App.tsx` renders `<StatusBar style="dark" />` (`expo-status-bar`) so icon/text color is always dark — the app has no dark mode, so it should never follow the system theme. That's a JS-level override that only takes effect once the bundle has mounted, though; before that (native splash screen, cold start), Android falls back to whatever `androidStatusBar` says in `app.json` — which was unset, so it used the OS default (light/white icons on some devices), explaining the inconsistent white-status-bar-on-launch report ("sometimes I see it in white, closing and reopening fixes it"). Fixed by adding an explicit `androidStatusBar: { barStyle: "dark-content", backgroundColor: "#FBF7F2", translucent: false }` block to `app.json` so the native baseline matches the JS override from the very first frame. **This is a native config plugin setting, not a JS/OTA change** — like the Sentry Gradle fix above, it's baked in at build time, so it needs a fresh `eas build`, not just `eas update`, before it takes effect on installed devices.

## Screen map

### Launch video — [src/screens/LaunchVideoScreen.tsx](src/screens/LaunchVideoScreen.tsx)
Plays [assets/applaunch.mp4](assets/applaunch.mp4) fullscreen (muted, no loop) before anything else renders — `RootNavigator` holds a `showLaunchVideo` flag and renders only this screen until it finishes, while `useBootstrapSession` keeps resolving in the background so the app already knows where to route once the video ends. Advances on `playToEnd`, on a player error, or after an 8s fallback timeout (never gets stuck); a "Skip" button in the corner does the same. Uses `expo-video` — a native-module dependency, so unlike most changes it needs a real `eas build` (not just `eas update`) to work on any already-installed standalone build; Expo Go and the local dev server pick it up immediately since Expo Go bundles the SDK's core native modules. **History**: this feature was removed once already (see git log for "Remove launch video") because shipping it via `eas update` before the corresponding native build existed crashed every already-installed app — any JS bundle referencing an unlinked native module crashes on load, not just video-related changes, so it blocked all OTA updates until pulled. Re-added with a new video; the fix is discipline, not code - never let an `eas update` referencing this screen go out ahead of the `eas build` that links `expo-video` natively.

### Onboarding — [src/screens/OnboardingScreen.tsx](src/screens/OnboardingScreen.tsx)
Single screen, three sequential steps driven by user state (not separate routes):
1. **AuthStep** — sign in / sign up (email+password) or "Continue with Google" (Supabase OAuth via `expo-web-browser`). Sign-up without email confirmation shows a "check your email" holding state. "Forgot password?" (sign-in mode only) requests a reset link (`requestPasswordReset`) and shows the same kind of holding state. **Requires `kinly://reset-password` to be added to Supabase Dashboard → Authentication → URL Configuration → Redirect URLs**, same as `kinly://auth-callback` was for Google sign-in — without it, `resetPasswordForEmail`'s `redirectTo` gets silently rejected. `signUp()` didn't set `emailRedirectTo` at all until a real user hit it — the confirm-your-email link redirected to the Supabase project's default Site URL (still `localhost`) instead of back into the app, leaving the device signed out even though the email was already confirmed server-side (confirmation happens at Supabase's `/verify` step, before the broken redirect). Fixed by passing `emailRedirectTo: Linking.createURL('auth-callback')`, same URL Google sign-in already uses (so it's already allow-listed) — `completeEmailConfirmation()` in [src/lib/auth.ts](src/lib/auth.ts) now handles that URL landing back in the app the same way `completePasswordRecovery()` handles reset links, calling `setSession()` so a freshly confirmed user is signed in immediately instead of being sent back to the login form.
2. **InterestsStep** — shown when `user.interests === null`; picks pillars (health/wealth/ideas/learning/relationships) via `InterestPicker`, saved with `useSetInterests`. `InterestPicker`'s chips were under the 44pt touch-target minimum (padding alone drove height) and relied on color alone to show selection (an active chip just flipped to `category.solid` background, no independent affordance) — both fixed together: `minHeight: 46`, bigger icon/text/padding, and a white `CheckIcon` (the new prop-driven one in `MonoIcons.tsx`, needed because the raw `.svg` import is hardcoded orange and would be invisible against a colored active chip) plus `accessibilityRole="checkbox"` so the multi-select nature (pick one or more, not a single radio choice) is conveyed by an icon and a screen-reader role, not just a color swap.
3. **CircleStep** — create a new circle (generates an invite code) or join one by code (`useCreateCircle` / `useJoinCircle`, both backed by `security definer` RPCs to sidestep RLS ordering issues). After creating, shows an `InviteStep` with a WhatsApp share button. Copy deliberately frames a solo start as valid, not a failure state ("Start solo, invite friends when you're ready") — both in the header subtitle and a caption under "Create Circle" — since a circle of one is a fully working, fully alive circle (garden, goals, celebrations all work at n=1; see Feature 2 in the Celebrations bullet below for how the app earns the first invite instead of demanding one upfront).
4. **Progress dots** — a small two-dot indicator (`StepDots`) renders above `InterestsStep`/`CircleStep` once a user exists, showing which of the two counted post-auth steps they're on; `AuthStep` itself isn't counted, since signing in is a gate before the flow, not a step within it. Previously there was no indication of how many steps remained at all.

### Today (home tab) — [src/screens/TodayScreen.tsx](src/screens/TodayScreen.tsx)
The app's landing screen — redesigned to have a clear "what should I do right now" purpose rather than being a raw activity feed.
- Personalized greeting + date ([src/lib/greeting.ts](src/lib/greeting.ts)).
- `MoodCheckinCard` — first card on the screen, deliberately: a check-in that needs zero goals to be useful (the one thing a brand-new user can always do). The card's title stays "How's today going?" in both states, deliberately — it used to switch to "How the circle's doing today" once checked in, which read as though the *whole card* had become about the circle, made the "Change" link ambiguous (change what?), and didn't match what the grid actually shows (a roster of individual moods, not a computed circle-wide mood). Now the title never changes; a `MOOD_SENTENCE`-driven line ("You're feeling great today") replaces it as the personal status line once checked in, and the circle grid gets its own small "Circle check-ins today" caption instead of borrowing the card's title. A "Change" link next to the status line reopens the picker to revise today's check-in. `mood_checkins` (migration `0023_mood_checkins.sql`, RLS mirrors `goals`' circle-membership pattern exactly) is unique per `(user_id, circle_id, checkin_date)`; changing your mind same-day upserts silently. Reuses the standard per-row nudge affordance for free — every event type already gets one, mood check-ins didn't need special-casing there.
  - **Feed sync**: a feed `events` row (`type: 'mood_checkin'`) used to be emitted only on the *first* check-in of the day and never touched again, so Circle Activity kept showing whatever mood was picked first even after changing it later — the grid bubble (a live query) and the feed text (a frozen snapshot) could disagree indefinitely. Needed a new `UPDATE` RLS policy on `events` (migration `0030_events_update_policy.sql`) — the table previously only had `SELECT` and two narrow `INSERT` policies, no `UPDATE` at all, scoped the same way (`user_id = auth.uid()`, may only touch your own rows).
  - **Atomicity** (migration `0031_atomic_mood_checkin.sql`): the first version of the fix above did the events write as a client-side read-then-insert/update (SELECT for today's existing event, then INSERT or UPDATE depending on the result) with no DB-level uniqueness guard backing it — two concurrent submissions (a double-tap, two devices) could both miss the existing row in their own SELECT and each insert a duplicate event. `mood_checkins` itself was already race-safe (a real unique constraint backs its upsert); `events` wasn't. Fixed by moving the whole read-modify-write into `submit_mood_checkin(p_circle_id, p_mood, p_tags)`, a `security definer` RPC that does both the `mood_checkins` upsert and the `events` upsert in one atomic call — the latter backed by a new partial unique index (`events (circle_id, user_id, checkin_date) where type = 'mood_checkin'`; `events.checkin_date` is a new nullable column, meaningful only for this event type). `useSubmitMoodCheckin` now just calls the RPC — no more client-side SELECT, no more `userId` parameter (the RPC identifies the caller via `auth.uid()` the same way `create_circle`/`join_circle_by_invite_code` already do).
  - Friends' check-ins already appear in Circle Activity without any special-casing — `useEvents` fetches every event for the whole circle, not just the caller's own, so once a circle-mate checks in their mood event shows up the same way a goal completion or streak would.
  - **Two-step picker** (`MoodPickerModal`, a bottom-sheet-style `Modal`): step 1 is the required one-tap mood (`MoodOptionCard` — a large stacked card per mood, `Great`/`Okay`/`Tough`, `Feeling Great`/`Doing Okay`/`Having a Tough Day` as the label) with a Reanimated fill/scale/icon-bounce on tap, saved immediately; step 2 (after a ~320ms delay so the tap animation is actually seen) shows optional context tags for that mood — `MOOD_TAGS`, a fixed predefined list per mood (`great`: Workout/Loved today/Celebration/etc., `okay`: Sleepy/Low energy/etc., `tough`: Lots of work/Exhausted/etc.), multi-select, saved on "Done". Text-only chips, deliberately no emoji or icons — most of these tags (`Loved today`, `Financial`, `Lonely`) have no matching icon in the app's custom set, and adding emoji here would undo the icon migration done everywhere else. `mood_checkins.tags` (migration `0029_mood_checkin_tags.sql`) is a plain `text[]`, not a separate tags table — the tag vocabulary is fixed for now, so there's nothing to join against. **Deliberately deferred**: user-created custom tags (a `+ Add Tag` flow) and any trend surfacing off the tag data (e.g. "you logged Lots of work 8 times this month") — both are separate features with their own storage/aggregation design, not bundled into this pass. Picking a *different* mood than what's already stored resets the tag selection (the two moods' tag lists don't overlap, so carrying old tags across wouldn't make sense); re-picking the *same* mood keeps its existing tags as the starting selection.
- `GardenTeaser` — big emotional summary card of the circle's health %, links to Circle tab.
- `TodayGoalsChecklist` — the signed-in user's own goals not yet logged today; tap to log progress, shows a celebration modal on completion/streak milestones.
- `QuickActionsRow` — three shortcuts (Check In → Goals, Ask Friends → Connection, Start Challenge → Circle).
- **Circle Activity** — day-grouped timeline of circle events (goal completions, streaks, reminders, asks, challenge completions, mood check-ins, streak saves, plain progress-photo logs) from `useEvents`. Each row supports one-tap emoji "nudges" (cheer/water/walk/workout/keep going/streak) which call `generateNudgeMessage` (Claude via Edge Function `smooth-responder`) then insert a nudge row and fan out a push notification via the `notify-circle` Edge Function — gated to `event.user_id !== userId` (fixed a real bug: the nudge row used to render on every event including your own, letting you nudge/cheer yourself and generate an AI-written encouragement message directed at your own post). `mood_checkin` is the one event type whose feed icon isn't static per-type — `TodayScreen`'s `eventIcon()` shows the actual mood emoji from the event payload instead of a placeholder, since 😊/😐/😞 carries real information a generic icon wouldn't. A `reminder` row (streak-at-risk, from `check-streaks-at-risk`) additionally shows a "💧 Water their streak" button when it's someone else's — see the "Streak saves" cross-cutting bullet below. Any event whose payload carries `photo_path` (goal_completed, streak, or the dedicated `progress_photo` type) renders a tap-to-view-full-screen thumbnail — see "Photo proof check-ins" below.

### Circle (dashboard tab) — [src/screens/CircleScreen.tsx](src/screens/CircleScreen.tsx)
Everything about "how is my circle doing," moved off Today to keep it uncluttered. Cards are split into two tiers so the accountability loop stays the visual priority instead of competing equally with lower-frequency extras:
- **Primary (always visible)**: `GardenCard` (full per-member plant growth, derived from `useGarden`, stages `wilted → seed → sprout → tree → bloom`), `BuddyCard` (accountability buddy pairing/check-ins, plus a "💧 Water [buddy]'s streak" button shown when their goal is in the one-day grace window — see "Streak saves" below), `ChallengesCard` (circle challenges + completion celebrations). All share the same flat card shell now (see "Card shell" below) rather than each getting its own pastel category color.
- **Secondary, behind a `DisclosureSection` ("More for your circle")**: `VisionBoardCard`, `MeetUpCard` (manual RSVP, no calendar integration), `CircleAICard` (AI insight, Edge Function `circle-ai-insight`), `WeeklyRecapCard` (AI weekly summary + shareable scorecard, Edge Function `weekly-recap` — see "Shareable weekly scorecard" below). Collapsed by default — tap to expand. `CircleAICard`'s "Try: {suggestedChallenge}" button creates a real challenge via `useCreateChallenge`, then confirms with haptics + inline "✓ Started" state and scrolls the screen back up to `ChallengesCard` so the result is actually visible — closes the loop between the AI suggestion and the primary card it feeds.
- **`CircleSwitcher`**: a horizontal chip row at the top of the screen, shown only once a user is in more than one active-membership circle (a single-item switcher would just be clutter for the common case). Tapping a chip calls `setActiveCircleId`, the same app-wide store value every other tab already reads — not a local-to-this-screen selection — so Today/Goals/Connection/Profile all follow it too, keeping the whole app consistent about which circle is "active." A faster, more visible alternative to `CircleSettingsScreen`'s own circle list/switcher, not a replacement for it. Deliberately v1-simple: no per-user ordering or a "top 5" limit — that's real added complexity (a new preference to store, a settings UI to manage it) that only earns its keep once someone is actually in enough circles for ordering to matter, which isn't true for most users this early.
- Header "⚙️ Settings" link → `CircleSettingsScreen`.

### Goals — [src/screens/GoalsScreen.tsx](src/screens/GoalsScreen.tsx)
- `GoalSuggestions` — AI/heuristic goal ideas based on interests (`src/lib/suggestions.ts`), tap to customize and add.
- `AddGoalForm` — title, numeric target, optional category chip.
- Goal list (`FlatList`) of `GoalCard`s: progress bar, streak badge, "Log progress" button, "⋯" menu for edit/delete.

## Cross-cutting mechanisms

- **Auth input validation & error normalization**: `src/lib/auth.ts`'s `signUp`/`signIn`/`requestPasswordReset`/`updatePassword` are the single chokepoint every auth call passes through (no other code calls `supabase.auth.*` directly for these), so validation lives there rather than in each screen. [src/lib/authValidation.ts](src/lib/authValidation.ts) defines Zod schemas (email/password/name — format, length, HTML-tag and control-character stripping) run before any Supabase call; a rejected input never reaches the network. [src/lib/authErrors.ts](src/lib/authErrors.ts) then maps every failure (validation-rejected or Supabase-rejected alike) to one generic, non-leaking message per context — `signIn` failures are always "Incorrect email or password" regardless of whether the email doesn't exist, the password is wrong, or the input was malformed to begin with, since a different message per failure path would itself be a side channel. `signUp` never confirms/denies an existing email; `passwordReset` always says "if that email is registered..." on both the request-error and success paths. The real Supabase error text (and the Zod rejection reason) still goes to `Sentry.addBreadcrumb` for server-side monitoring — detail is fine in logs, never in the UI. Note this validation lives in application code, not a real backend: there's no Express/Next-style server in this repo, and Supabase's GoTrue service (not this codebase) owns password hashing and storage entirely — `auth.ts` is the closest thing to a trust boundary this architecture has.
- **Password recovery**: unlike Google sign-in (which resolves entirely inside `WebBrowser.openAuthSessionAsync`), a reset link arrives by email and can open the app cold, so it needs a real global deep-link listener — `useAuthDeepLink` ([src/hooks/useAuthDeepLink.ts](src/hooks/useAuthDeepLink.ts)), mounted once in `RootNavigator`, covers both `Linking.getInitialURL()` (cold start) and `Linking.addEventListener('url', ...)` (warm). Any URL containing `reset-password` gets handed to `completePasswordRecovery()`, which calls `setSession()` with the link's tokens; that makes supabase-js emit a `PASSWORD_RECOVERY` auth event, which `useBootstrapSession` listens for to flip `useAuthStore`'s `passwordRecoveryMode` flag. `RootNavigator` checks that flag before its normal Onboarding/Main branching and renders `ResetPasswordScreen` instead when it's set.
- **Celebrations**: `MilestoneCardModal` is the single celebratory UI, triggered from goal completion/streak milestones (Goals, Today) and achievement badges (Profile). A user's very first goal log *ever* (across every goal, every circle) fires a distinct celebration instead — "Your garden has started growing 🌱 / Gardens grow faster with friends" — with the modal's Share button replaced by a real invite (`shareMessage`/`shareLabel` props on `MilestoneCardModal`, built from `inviteMessage()` with the circle's actual invite code) instead of generic achievement-share text. This is Feature 2's cold-start hook: rather than asking a brand-new solo user to invite friends upfront, the app waits for their first real moment of progress and frames inviting as the reward for it. Detection (`useLogGoalWithCelebration`'s `isFirstLogEver`) short-circuits to skip the check entirely once a goal has been logged before (the common case), and is checked last so it never overrides a real completion/streak celebration landing on the same log.
- **Push notifications**: Supabase Database Webhooks → `notify-circle` Edge Function fan-out to Expo push tokens, filtered by each recipient's per-category mutes.
- **AI features**: all go through Supabase Edge Functions (never called directly from the client with a raw API key) — nudge messages, weekly recap, circle insight, future self.
- **AI rate limiting** (migration `0028_ai_rate_limits.sql`): before calling Anthropic, `generate-nudge-message`, `circle-ai-insight`, and `weekly-recap` each call a new `increment_ai_usage(fn, max)` RPC, which atomically upserts a per-user/per-function/per-day counter (`ai_rate_limits`, keyed `user_id, fn, day`) and returns whether the caller is still under the cap — `security definer`, so it runs as the table owner but resolves `auth.uid()` from the caller's own JWT, same pattern as `delete_my_account()`. Previously nothing stopped repeated calls beyond `max_tokens` bounding a single call's output cost, and the client's 30s React Query `staleTime` (soft, resets every app reopen, easily bypassed). Caps: nudges 30/day (a deliberate per-tap user action, so generous), circle-insight and weekly-recap 20/day each (fire on every Circle-tab visit, not a deliberate action, so still generous but capped against a runaway loop). All three functions require "Enforce JWT verification" ON in the Supabase Dashboard — without a real caller identity the RPC can't attribute usage to a user and always returns false. Every call site already tolerates a failed/empty AI response gracefully (nudge falls back to a canned message, insight/recap simply don't render their AI line), so hitting the cap degrades a feature rather than breaking it.
- **Derived vs. stored state**: Garden health/plant stages are computed client-side from goals data (`useGarden`), not a separate table — keeps it always in sync with real goal activity.
- **Distribution (pre-Play Store)**: no Play Store listing yet, so invites can't just say "download Kinly." `inviteMessage()` ([src/lib/share.ts](src/lib/share.ts)) appends an Android APK download link read from `EXPO_PUBLIC_ANDROID_APK_URL` — the current internal-distribution build's install page (`expo.dev/.../builds/<id>`), which EAS hosts automatically for any `distribution: "internal"` build, no separate hosting needed. That URL is tied to a specific build ID and goes stale on every new native build (rare — JS-only changes ship via `eas update` to already-installed apps instead), so it has to be updated by hand in `.env` after each `eas build`, not automated.
- **Privacy policy**: [privacy.html](privacy.html), a page on the same GitHub Pages site as [index.html](index.html) (reachable at `https://sriharsha557.github.io/kinly/privacy.html` — no custom domain configured, don't assume one when linking to it), grounded in what the app actually collects (checked against `src/lib/auth.ts`, the Supabase schema, and every Edge Function's third-party calls rather than written generically). Linked from three places: the website footer, `ProfileScreen` (below Sign out), and `OnboardingScreen`'s sign-up form (Play Store review specifically checks for a policy link at account-creation time, not just somewhere in settings).
- **Account deletion**: self-service, from `ProfileScreen`'s "Delete account" link → `DeleteAccountModal` (type-to-confirm "DELETE", mirrors `CircleSettingsScreen`'s leave-circle confirmation pattern but stricter given the stakes). Unlike every other delete in the app, this is a genuine hard delete, not a soft delete — migration `0019_soft_deletes.sql` deliberately broke `profiles`' FK to `auth.users` (`on delete no action`) specifically so this path would have to be explicit rather than an accidental cascade. `delete_my_account()` (migration `0020_account_deletion.sql`, security definer) leaves every circle the caller is in via the existing `leave_circle()` (correctly handles ownership transfer) then hard-deletes their `profiles` row, cascading through everything rooted at it (goals, ask_posts, vision_items, achievements, events, nudges, future_letters, push_tokens). The `delete-account` Edge Function calls that RPC with the caller's own JWT (so `auth.uid()` resolves correctly), then separately calls `auth.admin.deleteUser()` with the service-role key to remove the actual `auth.users` row — the one piece no RPC can do, since only the Admin API can delete an auth user, and that key must never reach the client. Deploy note: unlike `check-streaks-at-risk`/`notify-circle`, this function must keep "Enforce JWT verification" **ON** — it must only ever act on the caller's own account.
- **Soft deletes** (migration `0019_soft_deletes.sql`): `profiles`, `circles`, `circle_members`, `goals`, `ask_posts`, and `vision_items` — the only tables ever an explicit `DELETE` target — use a `deleted_at` column instead. Every other table only ever loses rows via `on delete cascade` from these (ultimately rooted at `profiles`, whose FK to `auth.users` was changed from `on delete cascade` to `on delete no action`), so stopping physical deletes at these 6 tables stops it everywhere, without adding an unused `deleted_at` to tables nothing directly deletes. `is_circle_member()`, `enforce_circle_member_limit()`, `join_circle_by_invite_code()`, and `leave_circle()` were all updated to filter/set `deleted_at` correctly (a left member's row is deactivated, not gone, and rejoining reactivates it via `on conflict`). `ask_posts` has no general update policy by design (posts are immutable after posting — see `0009`), so its soft-delete goes through a dedicated `soft_delete_ask_post` RPC instead of opening one.
- **Human-gated circle membership** (migration `0022_join_approval.sql`): a forwarded invite code used to grant instant access; joining now lands in `circle_members.status = 'pending'` until an owner/admin approves. `is_circle_member()` (the function nearly every RLS policy on the schema calls) now requires `status = 'active'`, so this one change is what keeps a pending member out of everything circle-scoped — goals, events, asks, achievements, nudges, garden — with no per-table changes needed. Two narrower policies (`circles`/`circle_members` SELECT) additionally allow a pending member to see the circle's name and their own membership row specifically, so a "waiting for approval" screen has something to show and can detect the pending → active transition; nothing else opens up. `approve_member()`/`reject_member()` (owner/admin only) and `cancel_join_request()` (self-service withdrawal) are new security-definer RPCs. `leave_circle()`'s ownership-transfer selection was also fixed here — it previously had no status filter, so a pending member could theoretically have been promoted to owner. Rejoining after leaving lands back in `pending`, deliberately, not straight back to `active`. **Goal/event/garden/ask/achievement scoping to a single circle was already correct before this migration** — `goals.circle_id` and matching RLS have existed since `0001_init.sql`; don't assume a cross-circle data leak exists there without checking the actual schema first.
  - **Client**: `useCircleMembers`/`useMyCircles` now carry `status`/`membershipStatus`; call sites that render "the circle's members" (`useGarden`, `BuddyCard`, `GuessWhoCard`, `CircleSettingsScreen`'s roster) filter to `'active'` explicitly, since the RLS layer now returns pending rows too (an active member can see the whole roster, pending included, so the owner's approval UI has something to list). `CircleSettingsScreen` gates the invite code display to owner/admin (previously visible to every member) and adds a "Join requests" section above the member list. `RootNavigator` renders `PendingApprovalScreen` instead of `MainTabs` whenever the active circle's membership is pending — determined via `useMyCircles`, polled every 5s while pending (no realtime channel exists in this codebase; polling matches its all–React-Query pattern) so approval is picked up automatically without user action.
  - **notify-circle**: fixed a related bug found while implementing this — its `events` branch fanned out to every `circle_members` row with no status filter, which would have pushed circle-activity notifications to pending (unapproved) members. Also added two new webhook-driven branches: a join-request notification to the circle's owner/admins, and an approval notification to the joiner — both need their own Database Webhooks added manually (INSERT and UPDATE on `circle_members`), same manual-deploy pattern as this function's other three webhooks.
- **Card shell**: `GardenCard`, `BuddyCard`, `ChallengesCard`, `VisionBoardCard`, `MeetUpCard`, `CircleAICard`, `WouldYouRatherCard`, `GuessWhoCard`, `MoodCheckinCard`, `GoalSuggestions`' suggestion cards, `TodayScreen`'s Circle Activity event cards, `GoalsScreen`'s goal cards, `ProfileScreen`'s achievement badges, and `FutureSelfCard` all share one flat card treatment — `#FFFEFA` background, a 0.5px `#E4DFD1` border, `20px` radius, and a 3px left accent bar — instead of each getting its own pastel `categoryColors.X.bg` wash (or, for the last three, a plain `colors.surface` + generic drop `shadow` that never got converted the first time around). This replaced the original per-card-category-color pattern (Circle previously showed a different pastel per card: teal-ish for Garden, purple for Challenges, gold for Vision Board, **blue** for Meet Up, **pink** for Buddy/mood check-in/Would You Rather; Today's event feed showed a different pastel per event type, including yellow for mood check-ins/challenge completions) after direct feedback that it read as too many competing colors. Titles are `colors.shellTitle` (`#22281F` — not `colors.textPrimary`, a deliberately separate warm-ink token for this shell), secondary text `colors.shellSecondary` (`#7A7A6E`), links/accents `colors.primary`. Both were raw hex duplicated across ~12 files before a code-review pass flagged one instance (`TodayScreen`'s `eventText`) as bypassing the theme object — same root cause as the `cardShell` duplication above, so it got the same fix: two new tokens in `theme/colors.ts`, all 26 occurrences replaced. Inner elements that sat on a pastel card (e.g. `MoodCheckinCard`'s pick buttons, `VisionBoardCard`/`MeetUpCard`'s item rows, `TodayScreen`'s nudge/water buttons) were re-backgrounded to `colors.inputBg` so they stay visually distinct against the now-white shell instead of blending in. Event/entry type is now told apart by icon alone, not color — `TodayScreen`'s `EVENT_ICON` map and `LifeTimeline`'s `ENTRY_ICON` map both dropped their per-type `bg`/`text` fields entirely. **Not converted**: `GardenTeaser` (Today's hero card) — uses a deliberate `LinearGradient`, not a flat pastel, which is a different, intentional design choice, not the pattern being replaced; `WeeklyRecapCard` was the other gradient holdout but has since been converted (see the Gradient-card contrast note above). `ProfileScreen`'s 3 stat tiles (Goals done/Best streak/Friends helped) were flattened the same way — one shared `colors.inputBg`/`colors.primary` treatment instead of a different pastel each; the 4th "Circle Settings" tile stays solid `colors.primary` since it's an actual button, not a stat display. `GoalsScreen`'s category-picker chips are the one remaining per-category-color spot, and deliberately so — it's a selection state (chip flips to `categoryColors[key].solid` only when active, matching `InterestPicker`), not a static background; they were also icon-only with no visible label until a user asked what they were for (mobile has no hover), so they now show the category label too, same as `InterestPicker`'s chips. - **Progressive disclosure**: `DisclosureSection` ([src/components/DisclosureSection.tsx](src/components/DisclosureSection.tsx)) is the shared collapse/expand wrapper used to keep lower-frequency features (Vision Board, Meet Up, Circle AI, Weekly Recap on Circle; the two Light Moments games on Connection) out of the primary view without deleting them — used so the core accountability loop (goals, garden, buddy, challenges, ask friends) stays visually dominant.

Icons are custom components in [src/components/icons/TabIcons.tsx](src/components/icons/TabIcons.tsx) (`HomeTabIcon`/`PeopleTabIcon`/`GoalsTabIcon`/`ChatTabIcon`/`ProfileTabIcon`) reproducing the `assets/icons/*.svg` "friendly face" mark family as prop-driven `react-native-svg` primitives (like `Logo.tsx`) instead of raw SVG imports, since the bar needs to dynamically tint icons — a hardcoded-color SVG import can't do that. **`PeopleTabIcon` (Circle tab) is the one exception** — Circle is the one tab that maps directly onto the brand's own "two people together" mark, so it renders `assets/brand/logo-white-glyph.png` (the same asset `OnboardingScreen`'s header uses) via RN `Image`'s `tintColor` style instead of hand-traced `Svg` paths, recoloring the same way the sibling icons do via their `color` prop just through a different mechanism. - **Streak saves** (migration `0025_streak_saves.sql`): a circle member can grant one grace day to a friend who missed exactly one day — never themselves, at most once per person per week. Every rule (never-self, must-be-circle-mate, once-per-person-per-week, and the goal must actually be in the exact single-day grace window — `last_logged_date = current_date - 2`, stale enough that yesterday was missed but not so stale that one grace day can't save it) is validated inside `water_streak()` (security definer), not a general RLS INSERT policy — same pattern as `approve_member()`/`soft_delete_ask_post()` elsewhere in this schema. Honesty matters here: progress is never touched, only `log_goal_progress()`'s streak-continuation math changes — it now treats a `streak_saves` row covering the gap day as "logged" when deciding whether the streak continues or resets to 1, so a watered streak is real continuation, not a faked completion. `water_streak()` also writes an `events` row (`type: 'streak_saved'`, actor = the waterer) so the save shows up in Circle Activity like any other event and fans out through `notify-circle` normally. Surfaced in the two places friends already act: a "💧 Water their streak" button on `TodayScreen`'s `reminder` event rows (gated to `event.user_id !== userId` — a reminder about your own streak never shows it), and `BuddyCard` (finds a waterable goal among the buddy's own goals via `useGoals`, not `useGardenState`'s aggregate per-member stage, since a specific *goal* needs watering, not a person). `useHasWaterMark()` — see the Goals bullet above — is a cheap "has this goal ever been watered" signal for the 💧 badge, not a precise trace of whether the *current* streak run specifically includes a saved day.
- **Photo proof check-ins** (migration `0026_checkin_photos.sql`): an optional photo on a goal-progress log, never required, never nagged. `checkin-photos` is a **private** Storage bucket — unlike `avatars`/`vision-images` (public, security-through-obscurity-only URLs), a circle-mate's photo mid-progress gets real RLS-enforced access control, gated by `is_circle_member()` on the folder's `circle_id`. That means every display needs a signed URL (`useSignedCheckinPhotoUrl`, 1-hour TTL) rather than a static public one — `EventPhoto` (`TodayScreen`) resolves one per photo and shows a loading placeholder until it's ready. Path convention is `<circle_id>/<user_id>-<timestamp>.ext`, deliberately *not* `<user_id>/...` like the other two buckets, both so RLS can gate by circle and so account-deletion cleanup (below) can isolate one user's photos within a folder several circle members share. `useLogGoalWithCelebration` attaches `photo_path` to whichever event a log already produces (`goal_completed`/`streak`/the first-ever-log celebration) — or, since a plain progress log normally produces **no** event at all, inserts a new `progress_photo` one specifically so an attached photo always has somewhere to surface; a log with no photo behaves exactly as before, zero new events. Compression uses `expo-image-picker`'s own `quality` option (`0.5`, a touch more aggressive than avatar/vision-board's `0.6`), not `expo-image-manipulator` — that package was assumed "already SDK-bundled" by the spec that requested this feature but isn't actually an installed dependency, and adding it would mean a new native module and a full `eas build` for what image-picker's built-in compression already covers. Account deletion (`delete-account` Edge Function) captures the caller's circle memberships *before* `delete_my_account()` leaves them (the RPC soft-deletes those rows as part of the same call, so this is the last moment the caller's own client can see them), then lists and filters each circle's `checkin-photos` folder by `${userId}-` filename prefix to remove just that user's photos, leaving the rest of the circle's photos untouched.
- **Shareable weekly scorecard** (migration `0027_weekly_scorecard.sql`): `WeeklyRecapCard` gets a "Share" link that opens the OS share sheet with a plain **text** summary (goals completed, best streak, most-watered friend, circle health + its week-over-week delta, plus the existing AI highlight sentence) — never a captured image. `react-native-view-shot`-style screenshotting is a native module and isn't an installed dependency; every change this whole session ships via `eas update`, so image capture stays out of scope rather than forcing a native rebuild for one button (the original spec anticipated exactly this and explicitly allowed a text fallback). Nothing auto-posts — the share sheet only opens on that explicit tap. Circle health has no stored history to diff against otherwise (see "Derived vs. stored state" above), so `check-streaks-at-risk` (already scheduled daily via migration 0016) was extended to also snapshot every active circle's health into `circle_health_snapshots` on its existing run, reusing that cron instead of adding a second one just for this. **Fixed a real bug found while extending it**: the function used to `return` immediately whenever no goal happened to be at risk that day, which would have skipped the new snapshot pass entirely most days and left the scorecard's history full of gaps — the early return is gone, the snapshot pass now always runs.
- **Analytics** (migration `0024_analytics_events.sql`): no analytics SDK anywhere in this app — `analytics_events` is a minimal, purpose-built table for exactly one thing right now: measuring % of circles reaching 3+ members within 48h of creation (Feature 2's cold-start success metric). RLS enabled with only an INSERT policy and no SELECT policy at all, so nothing in the client can read it back — it's queried ad-hoc via the SQL editor, not surfaced in-app. [src/lib/analytics.ts](src/lib/analytics.ts)'s `logAnalyticsEvent()` is fire-and-forget (a failed analytics insert must never break the feature it's attached to) and separates `actor_user_id` (who performed the action, defaults to `auth.uid()`) from `subject_user_id` (who the event is *about*) — `member_joined` is logged from `useApproveMember`'s `onSuccess`, i.e. the *approving* owner/admin's session, not the joining member's own, so those two are deliberately different people. Convention: the circle owner is implicitly member #1 at `circle_created` time with no separate `member_joined` logged for them, so "reached 3 members" means 2 subsequent `member_joined` rows for that `circle_id`.
